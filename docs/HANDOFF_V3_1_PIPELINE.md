- 이미지→영상 생성 오디오는 켠다. 현재 코드도 `generate_audio=True`다.
- 비용과 대기시간이 큰 처리는 대본 전체를 한 번에 실행하지 않는다.

## v3.1의 핵심 계층

```text
Episode
└─ Scene: 장소·시간·사건이 연속인 이야기 단위
   └─ Clip: 영상 생성 1회 단위, 반드시 5~15초
      └─ Block: 클립 내부의 구도별 컷
```

- 씬은 5초일 수도 30초일 수도 있다.
- 씬이 15초를 넘으면 같은 씬 안에서 여러 클립으로 나눈다.
- 블록은 별도의 영상 API 호출 단위가 아니다.
- `대표 스틸 1장(+필요한 보강컷) + 클립 전체 블록 콘티 → 멀티샷 영상 1개`가 목표다.
- 현재 코드의 `샷 하나 → 스틸 하나 → 영상 하나` 구조는 목표 구조가 아니다.

## 확정된 실행 전략

### 화 전체에서 먼저 처리하는 1~4단계

이 단계에서는 이미지와 영상을 만들지 않는다.

1. 씬 설계
   - 장소 변경, 시간 경과, 사건 전환에서만 새 씬을 만든다.
   - 같은 장소·시간·사건이면 합치고 길이는 클립으로 해결한다.
2. 엘리먼트 메타데이터 정리
   - 인물·장소·의상·소품의 기존 등록명을 우선 재사용한다.
   - 이름/ID만 정리하고 레퍼런스 이미지는 아직 전부 만들지 않는다.
3. 씬 공통 선언
   - 등장인물과 인물별 의상, 장소, 무드/조명, 플롯 소품, 180도 액션 라인을 확정한다.
4. 클립 분할 및 러닝타임 검증
   - 모든 클립은 5~15초다.
   - 화 전체 기본 90초, 허용 범위 90~110초다.
   - 범위를 벗어나면 콘티 블록을 조작해 채우지 말고 대본 분량 문제로 보고한다.

### 5단계부터 씬 하나씩 완주

```text
씬1 상세 블록
→ v3.1 자동 검증
→ 씬1에 필요한 레퍼런스만 생성
→ 클립별 대표 스틸(+보강컷)
→ 사용자 승인/재생성
→ 클립별 멀티샷 영상
→ 씬1 완료
→ 씬2 시작
```

첫 구현은 완전 순차로 한다. 안정화 뒤에는 씬1 스틸 승인 후 씬1 영상 생성과 씬2 텍스트 콘티만
병렬화할 수 있지만, 연속성 상태가 확정되기 전 다음 씬의 이미지 생성은 시작하지 않는다.

## 씬 통과 조건과 상태 머신

씬은 아래 조건을 모두 만족해야 다음 씬으로 넘어간다.

- 상세 콘티 자동 검증 통과
- 필수 엘리먼트 레퍼런스 준비 완료
- 모든 클립의 대표 스틸 승인
- 모든 클립 영상 생성 성공

권장 저장 상태:

```text
pending
→ drafting
→ validating
→ references_ready
→ stills_ready
→ approved
→ video_generating
→ completed
```

각 단계는 멱등적이어야 한다. 서버 재시작 또는 실패 후 `completed` 씬을 다시 만들지 않고 현재
씬부터 이어가야 한다.

## 씬/클립/블록 생성 규칙 요약

### 씬 헤더

내부 Markdown 정본은 대략 다음 구조다.

```markdown
■ 씬2 · [편의점-카운터] (location_id) · 23초 · 수상한 제안
등장: 강태민(character_id / 의상: 세미정장A) · 이수진(character_id / 의상: 편의점유니폼A)
무드/조명: 차가운 심야 형광등, 낮은 채도, 청록색 그림자
소품: 현금봉투A = 흰색 서류 봉투, 약간 두꺼움, 봉인되지 않은 상태
라인: 강태민↔이수진 · 카메라 = 출입구 쪽 반원 → 프레임 좌우: 강태민=오른쪽/이수진=왼쪽

─ 클립 2-A · 8초 · 현금 봉투를 꺼내다 ─
[3초] 미디엄(허리 위)/아이레벨/반측면-좌/이수진 위주 — 자세: ... / 동작: ...
[2초] 인서트/아이레벨/인서트-계산대 위 현금 봉투 — 자세: ... / 동작: ...
[3초] 클로즈업(가슴 위)/로우/반측면-우/강태민 단독 — 자세: ... / 동작: ...
에셋: 스틸 (미정) · 영상 (미정) · 레퍼런스 (미정)
```

### 블록 분할

- 구도 변화만이 블록 분할 사유다.
- 구도는 `샷 크기/앵글/방향/대상` 네 값이다.
- 화자 교체와 동작 완결만으로 블록을 나누지 않는다.
- 일반 블록은 3~5초, 인서트는 1~2초다.
- 결정적 감정 순간만 6~8초를 허용하며 한 클립에 최대 한 번이다.
- 같은 단독 인물의 연속 블록은 샷 크기를 최소 2단계 벌리거나 방향/앵글을 크게 바꾼다.

### 자세와 동작

모든 블록은 시작 자세와 그 블록에서 생기는 변화를 분리한다.

- 행위 라벨
- 공간 기준 위치와 기준물까지의 거리
- 접지면
- 체위
- 왼손/오른손과 프레임 기준 위치
- 몸 방향
- 고개 방향
- 시선
- 인물 간 거리
- 프레임 안/밖
- 접촉 중인 타인의 손·팔·접촉 높이
- 소품의 소유자·위치·상태

블록 N의 종료 상태를 블록 N+1의 시작 자세에 그대로 복사한다. 클립·씬 경계도 같은 사건이
이어지면 릴레이를 유지한다.

### 카메라와 공간 방향

- 대화/대치 구간마다 180도 액션 라인을 선언한다.
- 인물 프레임 좌우, 보이는 얼굴 좌/우, OTS 걸침 어깨 위치를 라인과 맞춘다.
- 라인 넘기는 중립 컷, 화면에 보이는 카메라 무빙, 인물 이동 후 라인 재선언만 허용한다.
- 블록당 카메라 무빙은 하나이고 대상과 도착점을 명시한다.
- 롱샷/전신은 인물이 프레임 세로에서 차지하는 비율을 명시한다.

### 소리

- 대사는 대본과 글자 그대로 같아야 한다.
- 화면 속 입이 보이며 말하면 `(립싱크)`다.
- 같은 장소의 프레임 밖 목소리는 `off`다.
- 내레이션/속마음은 `V.O.`이며 화면 속 입이 움직이면 안 된다.
- 목표 편집 구조에서는 모든 `caption`을 나레이션으로 읽지 않는다.
- 립싱크·현장음은 생성 영상 오디오를 사용하고, V.O.는 별도 TTS/편집 레이어 후보로 분리한다.

### 안전

- 직접 폭력·과도한 밀착·노출은 콘티 단계에서 안전한 시각 표현으로 바꾼다.
- 감정은 유지하되 타격 직후 리액션, 소품 인서트, 안전 명시 괄호로 처리한다.
- 예: `목걸이를 뜯는다`가 아니라 `낚아채 손에 쥔다(체인 온전, 목 당김 없음)`.

## 스틸과 영상

### 스틸

- 각 클립의 감정·사건을 가장 잘 요약하는 블록의 시작 상태를 대표 스틸로 고른다.
- 클립당 대표 스틸 1장이 기본이다.
- 메시지, 반지, 목걸이 같은 핵심 인서트 또는 크게 다른 감정 구도가 필요할 때만 보강컷을 만든다.
- 씬 공통 선언, 선택 블록의 자세, 역할별 엘리먼트 참조를 합쳐 이미지 프롬프트를 만든다.
- 같은 씬의 이전 승인 스틸을 연속성 앵커로 전달한다.

### 영상

- 영상 호출 단위는 클립이다.
- 입력은 대표 스틸과 클립의 전체 블록 콘티다.
- 각 블록의 시간, 구도 전환, 자세, 동작, 카메라, 대사/립싱크/off/V.O.를 전달한다.
- 9:16, 생성 오디오 활성화다.
- 한 클립 실패 시 해당 클립만 재시도하고 다른 완료 클립은 보존한다.
- 보강 스틸을 영상 모델에 직접 다중 참조로 넘기려면 현재 단일 PNG 영상 래퍼 확장이 필요하다.

## 씬 간 연속성 핸드오프

완료 씬은 다음 씬에 아래 상태를 넘긴다.

```json
{
  "wardrobe": {"이수진": "편의점유니폼A", "강태민": "세미정장A"},
  "props": {"현금봉투A": "강태민의 오른손에 든 상태"},
  "location": "편의점-카운터",
  "lighting": "차가운 심야 형광등",
  "character_state": {
    "이수진": "계산대 안쪽에 서 있음",
    "강태민": "출입문 방향으로 돌아선 상태"
  },
  "last_approved_still": "scene1/clip1-B/still.png"
}
```

`직후` 연결이면 상태를 그대로 상속한다. 시간/장소가 바뀌면 얼굴 같은 영속 요소만 유지하고
자세·소품·조명을 새로 선언한다.

## 현재 코드에 이미 구현된 것

다음은 `4553fc3`에 구현되어 있다.

- 콘티의 인물→의상 매핑 추출 및 구버전 샷 복원
- 공백/하이픈 표기만 다른 의상 레지스트리 중복 완화
- 샷 JSON의 `costumes`, `continuity`, `current_shot` 필드 요구
- 인물 얼굴 참조와 의상 참조 역할 분리
- 의상 참조를 특정 인물에게만 적용하도록 프롬프트 고정
- 사용자 제공 `CONTINUITY REQUIREMENT` 템플릿을 이미지 프롬프트에 적용
- 전체 컷 생성 경로에서 직전 스틸을 다음 컷의 연속성 이미지로 전달
- 기존 컷 재생성 시 가장 가까운 이전 스틸을 연속성 앵커로 사용
- 이미지→영상 `generate_audio=True`
- 미리보기는 씬마다 대표 샷 한 장만 생성

## 현재 코드와 목표 구조의 차이

현재 구현은 아직 아래 구조다.

```text
대본 → 씬 설계 → 상세 콘티 → 씬별 shot JSON
→ 대표 shot 스틸 미리보기
→ 전체 모드에서는 shot마다 스틸 생성
→ shot마다 영상 생성
→ shot 영상 합본
```

주요 현재 함수는 `pipeline/orchestrator.py`에 있다.

- `generate_scene_plan`
- `generate_conti`
- `generate_shots_by_scene`
- `_representative_shot`
- `_cut_stills_from_shots`
- `generate_image_for_shot`
- `generate_cuts_for_scene`
- `generate_video_for_cut`
- `produce_episode_video`

기본 환경값 `DEMO_FIRST_CUT_ONLY`가 `1`이면 미리보기 대표 스틸 하나만 영상화하고 합본하지 않는다.
`0`이면 현재의 모든 shot을 각각 이미지→영상화한 뒤 합본한다.

목표 개편에서는 `shots_by_scene` 중심 구조를 `scenes[].clips[].blocks[]`로 전환해야 한다.

## 권장 구현 순서

1. v3.1 규칙을 저장소 내부 프롬프트/명세로 편입한다.
2. `Scene`, `Clip`, `Block`, `SceneHandoff`, 상태 머신 스키마와 정규화 함수를 만든다.
3. 화 전체 1~4단계 생성기와 결정론적 러닝타임/구조 검증기를 만든다.
4. 씬 하나의 상세 블록 생성 및 v3.1 검증/부분 재생성 경로를 만든다.
5. 기존 요소 레지스트리를 씬별 지연 레퍼런스 생성 방식으로 연결한다.
6. 대표/보강 스틸 선택과 생성 단위를 scene-shot에서 clip-anchor로 바꾼다.
7. 영상 래퍼와 저장 경로를 `scene_num/cut_num`에서 `scene_num/clip_id` 중심으로 확장한다.
8. 클립 전체 블록을 멀티샷 모션 프롬프트로 변환한다.
9. 씬 완료 상태와 다음 씬 연속성 핸드오프를 저장한다.
10. 합본에서 기존 `caption` 자동 나레이션을 제거하고 오디오 층을 분리한다.
11. 기존 저장 프로젝트 마이그레이션 또는 안전한 폴백을 추가한다.
12. 프런트에서는 내부 콘티를 숨기고 현재 씬의 대표 스틸 승인/재생성/진행 상태만 보여준다.

큰 개편이므로 한 커밋에 전부 넣지 말고, 스키마·텍스트 생성/검증·스틸·영상·UI 단위로 나누어
각 단계마다 기존 프로젝트를 깨뜨리지 않는 테스트를 실행한다.

## 최소 인수 조건

- 23초 씬이 5~15초 범위의 복수 클립으로 분할된다.
- 클립/블록/씬/화 시간 합 검증이 결정론적으로 실패를 잡는다.
- 씬1이 완료되기 전 씬2의 이미지·영상이 생성되지 않는다.
- 서버 재시작 후 완료 씬을 건너뛰고 현재 씬부터 재개한다.
- 클립 하나당 대표 스틸 한 장만 기본 생성한다.
- 보강컷은 명시된 블록에만 생성한다.
- 클립 하나당 영상 API 호출 한 번이 기본이다.
- 영상 프롬프트에 모든 블록 시간·구도·동작·소리 층이 포함된다.
- 같은 씬의 의상·장소·조명 선언이 모든 클립에서 동일하다.
- `직후` 씬은 이전 씬의 의상·소품·상태를 상속한다.
- 사용자 API/UI 응답에 내부 콘티와 생성 프롬프트가 노출되지 않는다.
- 기존 대표 스틸과 영상 생성 경로는 마이그레이션 기간 동안 안전하게 동작하거나 명확히 변환된다.
